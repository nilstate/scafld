# scafld 1.7.0 Cutover

Created: 2026-04-28
Status: planned

## Why this cutover

scafld 1.x works for the daily spec → build → review → complete loop, but
dogfooding has surfaced a small set of structural pains the
patches-and-flags surface can't fix without compounding cruft. None
justify a major version, but together they justify a focused cutover
across 1.6.x patches culminating in 1.7.0.

The pains:

- **Acceptance criteria advance phases when the work hasn't been done.**
  `scafld build` calls `compileall + unittest`, both pass, phase marks
  complete — even when the spec's actual implementation work is missing.
  Lenient substring matching in
  [`acceptance.py:168-202`](../scafld/acceptance.py) is the pattern; the
  missing structural-result parse is the gap.

- **Hand-edited review files bypass the challenger gate.** `scafld
  complete` reads the review file but doesn't verify it came from a real
  challenger. The packet sha256 is already computed in provenance — we
  just don't check it.

- **Spec validation is duplicated.** `.ai/schemas/spec.json` exists as
  Draft-07 JSON Schema and
  [`lifecycle_runtime.py:69-134:validate_spec`](../scafld/lifecycle_runtime.py)
  is a hand-written Python validator that re-implements a subset of
  the same rules. Drift is inevitable.

- **Review is the only operation that takes minutes, and it's fully
  sync.** Operators and agents both have to babysit it; agents resort to
  shell backgrounding. The watchdog/transient-retry/structured-output
  hardening lives inside a single blocking call — the async surface to
  expose those guarantees doesn't exist.

- **Provider differences are scattered.** codex and claude live in
  parallel arglist builders, parallel env setups, parallel default
  models, parallel hint regexes, parallel selection-reason strings.
  Adding a new provider is a 5-place edit; removing one is a 5-place
  audit.

- **Multi-spec conflicts surface only at review time.**
  [`audit_scope`](../scafld/audit_scope.py) classifies overlap during
  `scafld review` but not during `scafld plan` or `scafld build`. We
  hit this repeatedly while running parallel specs and had to retro-edit
  `ownership: shared` annotations.

- **Operator UX accumulates noise.** The "fell back to weaker Claude
  isolation" warning fires every run on machines without codex. Error
  messages name symptoms ("provider exited with 1") rather than causes.
  None are bugs; together they become fatigue.

These aren't bugs. They're the natural outcome of a 1.x design that
solved the loop and accreted hardening as it found pain. 1.7.0 makes a
small set of structural moves and leaves the rest of scafld intact.

## What this is, and isn't

## The 10/10 target

The model 1.7.0 ships toward:

```
scafld plan my-task -t "..." --command "..."     # one task description
scafld build my-task                              # run, validate
scafld review my-task                             # one round, severity-gated
scafld complete my-task                           # ship
```

An LLM writes one task description, runs four commands, ships clean
work. No 200-line yaml ceremony. No 4-6 review rounds. No
scope-conflict surprises. No fake-gate hand-edit vector. No
"compile + unittest pass with no real work" phase advance.

**Is:** the four structural changes that hit every 10/10 criterion
(slim plan, single-round review, scope-conflict-at-plan, hand-edit
gate, evidence-required acceptance). Ships across 1.6.x patches and
one 1.7.0 minor.

**Isn't:** a redesign. The spec / session / handoff vocabulary stays.
Plan / approve / build / review / complete stays. The CLI surface stays
recognizable. Existing 1.x specs and workspaces continue to parse
(slim format is a strict subset of verbose; extra fields ignored). No
new required runtime dependencies.

## The 4 specs

Ordered by dependency. Each ships independently as a 1.6.x patch;
1.7.0 cuts the version once all four are landed.

1. **`runtime-rigor-acceptance`**
   Acceptance criteria gain `expected_kind` (`exit_code_zero`,
   `exit_code_nonzero`, `no_matches`) and `evidence_required: true`.
   Strict-only — undeclared `expected_kind` fails loudly without
   running the command. Hard cutover; existing 1.6.x specs with
   legacy `expected:` strings auto-resolve when they map cleanly,
   else fail with a one-line guidance error. Replaces lenient
   substring matching in [`acceptance.py:168-202`](../scafld/acceptance.py).

2. **`runtime-rigor-severity-gates`**
   First-round review verdict gates `complete` by default. The flow:

   - `scafld review` runs once. Returns verdict + findings.
   - `scafld complete` blocks **only** on `blocking` findings
     (high/critical severity).
   - Medium and low findings are advisory output. They don't block
     `complete` and they don't trigger re-review.

   **One round in the typical case.** A `pass` or `pass_with_issues`
   verdict (no blocking findings) → ship. One round, done. The
   rabbit hole — "review finds 3 things → fix → review finds 2 more
   → fix → review finds 1 more" — stops because medium/low don't
   drive iteration.

   **Iteration only when blocking findings exist.** Blocking →
   `complete` refuses → agent fixes → agent re-runs `scafld review`
   to verify → new round. Iteration is bounded by `blocking`
   severity, not by the existence of any findings.

   **Optional**: agent may voluntarily address medium/low findings
   as follow-up specs (or re-run review on the same task) — opt-in,
   not the expected loop.

   Operators who want strict iteration set
   `review.gate_severity: medium` to make medium findings block
   too. Default is `blocking`. The stopping rule moves from
   agent-discipline doc → CLI behavior.

3. **`runtime-rigor-gate-integrity`**
   `canonical_response_sha256` is already computed in provenance and
   already written into the review markdown's metadata block; the
   canonical packet body is already persisted to
   `.ai/runs/<task>/review-packets/review-<n>.json`. What's missing
   is the verify-on-complete step. Add `verify_review_seal(metadata,
   packet)` to `review_packet.py`; in `cmd_complete`, load the packet
   artifact, recompute, refuse on mismatch with
   `EC.REVIEW_GATE_BLOCKED`. Hard cutover: pre-1.7 review files (no
   `canonical_response_sha256` in metadata) fail; operator re-runs
   review under 1.7.0. Catches the hand-edit fake-gate.

4. **`runtime-rigor-clean-plan`**
   `scafld plan <id> -t "..." --command "..." --files "a.py,b.py"`
   produces a complete ~30-line spec. No `TODO` markers (so
   `validate_spec` passes cleanly without a fill round). One phase,
   one criterion with explicit `expected_kind: exit_code_zero` (so
   `evaluate_acceptance_criterion`'s strict-unset-reject doesn't
   fire). Verbose template path is preserved when `--command` is
   omitted, so multi-phase complex specs stay verbose by default.

   Plus: `audit_scope.classify_active_overlap` runs after scaffold;
   auto-tags overlapping files as `ownership: shared`; refuses with
   `EC.SCOPE_DRIFT` on exclusive conflict. No review-time
   `scope_drift` surprises.

   Schema-required fields not enforced by `validate_spec`
   (`task.context`, `task.objectives`, `task.touchpoints`,
   `task.acceptance`) are omitted by the slim template; renderers
   degrade gracefully (`handoff_renderer.py:115`,
   `output.py:269`, `projections.py:109` all use `or []` fallbacks).
   For complex tasks the operator extends the spec with the verbose
   blocks.

## Diminishing-returns stopping rule

scafld's adversarial review is non-deterministic and finds new edge
cases on every round. Treating "every finding must be fixed" as a hard
rule turns review into a never-terminating loop. The stopping rule:

- **Critical / high findings**: never skippable. Fix and re-review.
- **Medium findings**: judgment call. Fix when the cost is bounded;
  defer to a follow-up spec when the value is marginal. Record the
  decision in the spec's `planning_log` before `scafld complete`.
- **Low findings**: skippable when judged non-essential bloat
  (test-mock fragility, doc cosmetic drift, naming nits, CI flake
  risk on real-clock tests, lockstep observations between spec text
  and post-implementation refinements). Record skipped findings + the
  rationale in `planning_log`.

The agent owns this judgment. The CLI does not enforce it; the audit
trail is the entry in `planning_log` recorded before complete.

## 1.7.x roadmap (post-1.7.0)

These specs were considered for 1.7.0 and deferred — they don't move
the 10/10 needle directly. Drafts stay in `.ai/specs/drafts/`. Each
ships as a 1.7.x patch when its pain re-emerges.

- **`runtime-rigor-review-async`** — `scafld review --detach`,
  `scafld wait`, `scafld cancel`. The shell-`&` pattern is enough
  for now; pick up when operators or agents hit a real workflow that
  needs daemonized review (e.g., CI orchestration, multi-spec
  coordination).
- **`runtime-rigor-schema-unification`** — `jsonschema` as soft
  dependency. Pick up if spec/packet drift bugs keep surfacing
  post-1.7.0.
- **`runtime-rigor-provider-registry`** — collapse codex/claude
  conditionals into a Provider protocol. Pick up when a third
  provider is added.
- **`runtime-rigor-verdict-extensions`** — accept `pass - <note>` in
  the verdict field. Pick up if operators ask for it.
- **`runtime-rigor-polish`** — once-per-version warnings, cause-named
  errors, custom-YAML-parser retirement, dual-path-resolver cleanup.
  Pick up as a maintenance batch when the rough edges add up.

The watchdog suspend bug surfaced during cutover dogfooding is fixed
inline in `_provider_watchdog` (the 6-line `_watchdog_elapsed(start_wall,
start_mono)` helper using `max(wall_elapsed, monotonic_elapsed)`).
No spec ceremony needed; ships alongside the acceptance work.

## Cutover by area

### Acceptance rigor

Replace `acceptance.check_expected` substring matching with structured
acceptance contracts. Each criterion declares an `expected_kind`:

```yaml
- id: "ac1_1"
  type: "test"
  command: "python3 -m unittest tests.test_review_runner"
  expected_kind: "exit_code_zero"
```

Kinds: `exit_code_zero`, `exit_code_nonzero`, `no_matches`, `regex`
(with explicit pattern), `json_path`. The legacy `expected: "exit
code 0"` etc. continue to work, mapping to the new kinds.

Add `evidence_required` flag. When true, criterion-pass requires
non-empty stdout or a referenced output artifact. Stops the "compile
+ unittest pass with no real work" pattern.

Add `acceptance_strictness` workspace setting (`lenient` | `strict`).
1.7.0 default: `strict`. Operators upgrading get a one-time `scafld
update` note.

### Gate integrity

`scafld review` already computes `canonical_response_sha256`. Persist
that hash into the review markdown's metadata block AND into a sibling
`sealed.json` containing the canonical ReviewPacket as the provider
returned it. `scafld complete` re-parses, recomputes, refuses to
advance if they differ.

`--runner local` and `--runner manual` paths produce a sealed packet
the same way — sealing party recorded as `local_runtime` or
`human_operator`. Hand-editing turns silent fakes into loud failures.

### Schema unification

`jsonschema` becomes a soft dependency: present in scafld's venv,
optional in operator environments. When importable,
`validate_spec` delegates to `jsonschema.validate(...,
schema=resolve_schema_path(...))`. Same for ReviewPacket. Hand-written
fallback stays as defense for constrained installs.

### Provider registry

```python
class Provider(Protocol):
    name: str
    binary_env_var: str
    default_models: tuple[str, ...]
    default_timeout: int
    isolation_level: str

    def build_args(self, *, model, schema_path, schema_json,
                   output_path, session_id) -> list[str]: ...
    def subprocess_env(self) -> dict[str, str]: ...
    def parse_stdout(self, stdout: str) -> ParsedProviderOutput: ...
```

`PROVIDERS = {"codex": CodexProvider(), "claude": ClaudeProvider()}`.
Internal refactor; no operator-visible changes.

### Review async

```bash
scafld review <task>          # blocks at TTY (1.x); detaches at non-TTY
scafld review <task> --detach # always returns run_id and exits
scafld wait <task>            # blocks until terminal state, prints verdict
scafld cancel <task>          # writes cancel marker; watchdog observes
```

Daemon child runs the existing `_run_external_review_once` body. Parent
prints `{run_id, pid, schema_arg_attached, ...}` and exits. Daemon
writes `provider_invocation` entries to `session.json`.

`scafld status` already reflects the running invocation.

### Early conflict detection

Add `audit_scope.classify_active_overlap` calls at:

1. `scafld plan` — warn if new spec's declared files conflict with any
   active spec.
2. `scafld approve` — block on unresolved exclusive conflicts.
3. `scafld build` start — same check before snapshotting baseline.

No new logic; refactor of existing audit code.

### Operator UX

- `_external_review_warnings` adds branch for
  `selection_reason == "codex_unavailable"`: emit `note:` (not
  `warning:`) once per scafld version. Stored in
  `.ai/runs/.notes-emitted.json`.
- Acceptance timeout names criterion + spec field: `criterion ac1_2
  (python3 -m unittest ...) timed out after 600s (set
  acceptance.timeout_seconds in spec)`.
- Default `fallback_policy` stays `warn` but text names cause + fix.
- `next_action` in JSON output always includes literal command string.

### Cleanup

- Drop custom YAML parser in
  [`config.py:14-70`](../scafld/config.py). Mandate PyYAML.
- Retire dual-path asset resolver
  ([`runtime_bundle.py:84-89`](../scafld/runtime_bundle.py)). Workspaces
  built on 1.x are migrated by `scafld update` to use only the managed
  `.ai/scafld/` paths. Legacy `.ai/` paths removed as resolution
  targets after one minor version of warning.
- Consolidate per-field YAML reads
  ([`spec_parsing.py`](../scafld/spec_parsing.py)) onto a single
  cached `load_spec_document` call.

## Release notes

Behavior changes that affect existing workspaces:

- **`acceptance_strictness` defaults to `strict` in 1.7.0.** First
  `scafld build` after upgrade prints a one-time note. Operators who
  want 1.x behavior set `acceptance_strictness: "lenient"` in
  `.ai/config.yaml`.

- **Review file integrity check (1.6.x).** First `scafld complete`
  after upgrade verifies the seal hash. If a workspace's review file
  predates the seal (no hash recorded), `complete` accepts it once
  with a loud one-time warning and stamps a fresh hash. Subsequent
  edits fail.

- **`--detach` flag is new (1.6.x).** Existing scripts that call
  `scafld review <task>` keep working at TTY.

- **No spec format change.** Existing `.ai/specs/**/*.yaml` parse and
  run unchanged.

- **`--runner local` / `--runner manual` keep working.** The
  integrity seal extends to those paths.

`scafld update` from 1.6.x to 1.7.0 is a bundle rewrite plus the
config-default note. No spec migrations, no session.json migration.

## Order of operations

The cutover is built as a sequence of 8 specs, each individually
reviewable and revertable:

1. `runtime-rigor-acceptance` — kinds, evidence, strictness (1.7.0).
2. `runtime-rigor-gate-integrity` — review packet seal (1.6.x).
3. `runtime-rigor-schema-unification` — jsonschema soft adoption (1.6.x).
4. `runtime-rigor-provider-registry` — codex/claude classes (1.6.x).
5. `runtime-rigor-review-async` — `--detach`, `wait`, `cancel` (1.6.x).
6. `runtime-rigor-early-conflict-detection` — audit at plan/build (1.7.0).
7. `runtime-rigor-verdict-extensions` — pass-with-note syntax (1.6.x).
8. `runtime-rigor-polish` — UX + YAML/dual-path cleanup (1.7.0).

Each spec lands on `main` as the next 1.6.x patch (or 1.7.0 for the
two minor-marked items). Strictness flip and conflict-detection at
plan time are the two operator-visible defaults that warrant a minor
bump.

## What we're not building

For clarity:

- Not a hosted UI, not a multi-user collab layer.
- Not a different vocabulary. Spec / session / handoff stay.
- Not a different review philosophy. Adversarial gate stays the core.
- Not a streaming event log. The async lifecycle is enough.
- Not provider abstractions beyond codex / claude / one or two future
  providers.
- Not async everywhere. Build/exec/complete stay sync; they're seconds.
- Not removing `--runner local` / `--runner manual`. They stay as
  documented degraded paths; the integrity seal makes hand-edits
  detectable, which is the actual fix.

This cutover leaves more of scafld unchanged than it changes. The
point is to remove the structural pain we observed in 1.x dogfooding
without inventing problems.
